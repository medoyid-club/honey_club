# Vision: слоёный профиль личности

Документ фиксирует долгосрочное видение системы личностных профилей в «Клубе медоедов»: зачем она нужна, как развивается по горизонтам и на каких принципах строится. Это не ТЗ — это рамка, к которой возвращаемся при принятии продуктовых и архитектурных решений.

## 1. Исходная идея

Личность можно описать как стек слоёв (по аналогии с моделью OSI):

- **нижние слои** — базовые черты (Big Five), почти не меняются;
- **средние** — характерологические радикалы, когнитивные стили (MBTI-подобные), меняются медленно;
- **верхние** — текущее состояние (энергия, настроение, стресс) и социальные роли, меняются постоянно.

Из метафоры следуют два инженерных принципа:

1. **Разделение по скорости изменения.** Черты, состояния и роли — разные данные с разными циклами обновления, хранением и политиками доступа.
2. **Тесты — это «протоколы», а не сама личность.** Big Five, радикалы, MBTI — взаимозаменяемые способы заполнить один и тот же профиль. Набор тестов должен меняться без ломки данных.

Главный сдвиг: **ценность не в уровнях, а в отношениях и взаимодействиях между людьми.** Уровни — статика и способ визуализации; продукт — динамика.

## 2. Архитектурная формула

> **Иммутабельные тесты → плоский вектор черт → поток состояний → визуальные слои.**

- **Сырое неизменяемо.** Каждое прохождение теста — иммутабельная запись (`assessments`): инструмент, версия, ответы, вычисленные шкалы, дата.
- **Профиль — производное.** Текущий вектор черт (`trait_profiles`) пересчитывается из тестов. Ярлыки («ENFP», «гипертим») вычисляются на лету из чисел и не хранятся как истина.
- **Черты — плоский вектор с неймспейсами** (`bigfive.openness`, `radical.hyperthymic`, `state.energy`), а не структура уровней. Философская модель уровней живёт в конфиге/презентации и может меняться без миграций.
- **Состояние — отдельный поток событий** (`state_checkins`), не поле профиля.
- **Социальная надстройка** (команды, роли, карта) — отдельный домен, который *читает* профиль, а не живёт в нём.

В пределе это event sourcing: всё — иммутабельный поток событий (тесты, чек-ины, эпизоды взаимодействий), профиль — проекция потока на текущий момент. Фундамент не ломается — добавляются только новые типы событий и новые проекции.

## 3. Типология взаимодействий

OSI-метафора описывает **статику** (что храним, как слоим, кому видно). Динамику описывает вторая метафора — универсальная модель систем на основе Стандартной модели элементарных частиц. Она даёт таксономию событий: у каждого события в потоке есть «тип взаимодействия», который определяет, *что ему разрешено менять*.

| Физика | В системе | Что это значит |
|---|---|---|
| **Кварки** | Вектор черт (`trait_profiles`) | Базовые характеристики. Как кварки не наблюдаемы поодиночке, так и черты видны только в связанных комбинациях |
| **Адроны** (связанные кварки) | Радикалы, типы, архетипы | Составные «частицы» из черт: никто не видит Openness=70 — видят «гипертима». Храним кварки (числа), вычисляем адроны (ярлыки) на лету |
| **Сильное взаимодействие** (глюоны) | Правила композиции | «Склеивает» черты в целостность: деривационные правила, по которым из шкал собираются радикалы и архетипы. Не события, а часть проекции |
| **Слабое взаимодействие** | События, меняющие идентичность | В физике — единственная сила, меняющая «аромат» кварка. У нас — редкие события, сдвигающие нижние слои: большой курс, кризис, повторный тест спустя год. Только они имеют право обновлять `trait_profiles` |
| **Электромагнитное** (фотоны) | Поток повседневных событий | Сообщения, квесты, чек-ины — 99% трафика. Быстрые, частые, меняют только состояние, до кварков не достают |
| **Гравитация** | Контекст среды | Слабая в каждом событии, но действует на всех и постоянно: культура сообщества, сезон, локаль. Не отдельные события, а измерения контекста, прикреплённые ко всем событиям; видна в агрегатах и когортах |
| **Поля** | Среда, культура | Окружение, в котором живут все взаимодействия: сообщество, ценности, язык платформы |
| **Тёмная материя** | Необъяснённый остаток | Систематическое расхождение «предсказание профиля vs реальное поведение». Не шум, а сигнал неполноты профиля — инженерное воплощение принципа «ярлык — не клетка» |
| **Энтропия / время** | Деградация уверенности | Тест двухлетней давности — не то же знание, что свежий. Уверенность затухает со временем → система мягко предлагает перепройти |

Практический итог — правила доступа событий к данным:

- **EM-события**: частые, пишут только в поток состояний;
- **слабые события**: редкие, единственные могут обновлять черты;
- **сильное взаимодействие**: не события, а правила вывода ярлыков из чисел;
- **гравитация**: контекст, прикладываемый ко всем событиям;
- **тёмная материя**: метрика расхождения «профиль vs поведение», мера честности системы.

Это заранее отвечает на вопрос горизонта 4 («какие события имеют право менять профиль») и защищает нижние слои от шума повседневных взаимодействий. Метафора упорядочивает мышление, но схему данных не меняет — события просто получают типизацию.

## 4. Горизонты развития

### Горизонт 1. Профиль как артефакт обучения

Профиль рождается из курсов, а не как отдельная фича:

- прошёл урок (например, про радикалы) → разблокировался тест → заполнился слой профиля;
- профиль — «персональный прогресс» школы: курсы достраивают карту себя;
- первый цикл вовлечения: профиль ⇄ курсы (система рекомендует курс под пустой/слабый слой).

Решает холодный старт: тесты «в пустоту» проходят неохотно, как часть курса — охотно.

**MVP:** один тест (Big Five — самый валидный «нижний протокол»), таблица `assessments`, страница профиля со «слоёным пирогом», где заполнен один слой, а остальные — «предстоит открыть» (механика прогрессии).

### Горизонт 2. От узлов к рёбрам: взаимодействия как данные

Профиль — не документ человека, а узел графа. Ценность — в рёбрах:

- **Совместимость по слоям, а не одним числом.** Нижние слои — долгосрочная совместимость («уживёмся ли»), состояние — ситуативная («стоит ли взаимодействовать прямо сейчас»). Честнее и полезнее любого «вы совпадаете на 87%».
- **Переводчик сообщений** (метафора инкапсуляции «Привет»): не «что он на самом деле имел в виду», а «человек с таким профилем, скорее всего, прочитает твоё сообщение вот так; вот варианты ближе к тому, что ты хотел передать». Прикладная эмпатия как сервис.
- **Тренажёр коммуникации:** симулированные диалоги с «собеседником» заданного профиля — безопасная песочница, где видно инкапсуляцию/декапсуляцию смысла. Мостик между курсами и социальной частью.

Важно: интерпретации сообщений — генеративные вероятностные гипотезы, **не данные о человеке**. Не хранить как факт, не показывать как «он на самом деле имел в виду».

### Горизонт 3. Социальная RPG (Stage 2)

Карта, квесты, команды — слоёная модель раскрывается полностью:

- **Квесты адресуют конкретные слои:** физические активности, эмоциональные практики, интеллектуальные задачи, социальные миссии, ценностные «арки». Слой профиля = тип контента для рекомендации.
- **Команды собираются по комплементарности,** не по похожести: стабильные слои — кто кого уравновесит, состояние — кто сейчас в ресурсе, роли — кто закроет какую функцию.
- **Роли наблюдаются, а не декларируются.** Человек говорит «я лидер», но в трёх квестах подряд был медиатором — система это видит и уточняет верхние слои.

### Горизонт 4. Самообучающийся профиль

> **Тесты — prior, взаимодействия — evidence.** Профиль — постоянно уточняемая гипотеза о человеке, которую обновляет его реальное поведение в системе.

Маховик: тесты дают стартовый профиль → профиль улучшает матчинг и квесты → взаимодействия дают поведенческие данные → данные уточняют профиль → матчинг точнее. Ценен цикл, а не отдельные части.

Побочный эффект — профиль показывает **траекторию**, а не ярлык: «полгода назад твой стресс-слой выглядел так, сейчас — так». Для школы личностного роста это и есть продукт: видимое доказательство роста.

**Эволюция продукта:** зеркало (вижу себя) → переводчик (понимаю другого) → навигатор (нахожу своих и свои квесты) → спутник роста (система видит динамику и ведёт дальше). Каждый этап — самостоятельный продукт на одном фундаменте событий.

## 5. Принципы (на весь путь)

1. **Ярлык — не клетка.** Профиль всегда подаётся как «текущая гипотеза, которая меняется». Иначе система консервирует людей в типах — против миссии школы.
2. **Прозрачность выводов.** Всё, что система *вывела* о человеке из поведения (а не из теста, который он сам прошёл), он должен видеть и мочь оспорить или скрыть. На дальних горизонтах — вопрос доверия №1.
3. **Асимметрия видимости.** Чем глубже слой, тем уже круг доступа. Матчинг и команды работают на огрублённых производных; сырые данные не покидают владельца.
4. **Научная скромность.** Big Five валиднее MBTI и радикалов; все инструменты — равноправные «протоколы» с метаданными об источнике, но матчинг опирается на более валидные шкалы. Результаты подаются как зеркало/гипотеза, не вердикт.
5. **Приватность by design.** Психологический профиль — чувствительные данные (GDPR, близко к special category): видимость по слоям (всем / друзьям / только мне), право на полное удаление, сырые ответы — только владельцу.

## 6. Опорная схема данных (эскиз)

```sql
-- Сырые прохождения тестов: неизменяемые, версионированные
create table assessments (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references profiles(id),
  instrument text not null,        -- 'bigfive' | 'radicals' | 'mbti_like' | ...
  instrument_version text not null,
  answers jsonb,                   -- сырые ответы
  scores jsonb not null,           -- вычисленные шкалы {"openness": 55, ...}
  taken_at timestamptz default now()
);

-- Производный «текущий профиль»: плоский вектор черт с неймспейсами
create table trait_profiles (
  user_id uuid primary key references profiles(id),
  traits jsonb not null,           -- {"bigfive.openness": 55, "radical.hyperthymic": 0.8, ...}
  computed_at timestamptz
);

-- Эфемерное состояние («сессия»): отдельный поток событий
create table state_checkins (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references profiles(id),
  energy int, mood int, stress int,
  created_at timestamptz default now()
);
```

Дальше (горизонты 2–4) добавляются события взаимодействий (`interaction_events`) и новые проекции — без изменения фундамента.

## 7. Чего сознательно не делаем

- Не зашиваем 7–10 «философских» уровней в схему БД — они живут в презентации.
- Не храним ярлыки типов как источник истины — только числа, ярлыки вычисляются.
- Не показываем интерпретации сообщений как факты о намерениях людей.
- Не строим матчинг одним числом совместимости — только по слоям.
- Не используем сырые ответы тестов за пределами профиля владельца.
