# Техническое задание  
## Проект: обучающая платформа с социальной RPG-картой

## 1. Цель проекта

Создать веб-платформу для онлайн-курсов, лекций и семинаров, в которой пользователи одновременно являются участниками социальной игры. Пользователь проходит обучение, развивает профиль персонажа «Я», получает доступ к карте мира, общению, квестам и совместимым контактам по интересам, целям и типу личности. Платформа должна объединять образовательный контент, личные кабинеты, оплату, социальные механики и интерактивную карту в одном продукте. [millermedia7](https://millermedia7.com/blog/how-to-design-a-dynamic-website/)

## 2. Основная идея

Пользователь регистрируется на сайте, проходит профильный тест, заполняет данные о себе и получает «игровой» профиль. На основе профиля ему предлагаются:
- курсы и мероприятия,
- другие участники с совместимыми профилями,
- карта с псевдолокациями,
- квесты, флэшмобы и социальные активности,
- режимы для дружбы, отношений, бизнеса и работы.  

Платформа должна быть не просто сайтом курсов, а живой социальной средой, где обучение и саморазвитие встроены в игровой процесс. [arxiv](https://arxiv.org/abs/2409.18360)

## 3. Функциональные блоки

- Страница о проекте.
- FAQ.
- Контакты.
- Политика конфиденциальности и пользовательское соглашение.

### 3.2 Пользовательский кабинет

- Профиль пользователя.

- Выбор псевдолокации.
- Просмотр карты.
- Список совместимых людей.
- Чат / сообщения.
- Участие в квестах и событиях.
- История активности.
- Баллы, уровень, статус персонажа.

### 3.3 Образовательный модуль
- Каталог курсов.
- Карточки курсов.
- Запись на курс.
- Оплата.
- Доступ к материалам.
- Лекции, записи, семинары.
- Блог и публикации ведущих.
- Уведомления о новых материалах и событиях.

### 3.4 Карта и игровая среда
- Карта с регионами и локациями.
- Возможность скрывать реальную геолокацию.
- Псевдонимные названия стран и регионов.
- Отображение пользователя на карте по выбранной зоне.
- Квесты и флэшмобы в безопасном формате.
- Социальные точки взаимодействия.

### 3.5 Совместимость и рекомендации
- Профильные оси.
- Тип личности.
- Совместимость по целям.
- Совместимость по состоянию.
- Совместимость по социальной роли.
- Рекомендации людей, групп и мероприятий.

## 4. Рекомендуемая архитектура

Я советую **не делать проект чисто статическим**. Это должен быть динамический сайт с гибридным подходом: публичный контент может рендериться статически, а пользовательская часть — работать как полноценное приложение. [lovable](https://lovable.dev/guides/static-vs-dynamic-website)

### Рекомендуемая схема
- **Frontend публичного сайта:** современный веб-фреймворк с возможностью SSR/SSG.
- **Backend:** Python API.
- **База данных:** PostgreSQL + PostGIS.
- **Карта:** Mapbox / Leaflet / OpenStreetMap.
- **Файлы и медиа:** object storage.
- **Чат и live-статусы:** WebSocket.
- **Платежи:** внешний платежный провайдер.
- **Админка:** отдельный интерфейс управления.

## 5. Рекомендованный стек

### Для MVP
- **Backend:** FastAPI или Django REST.
- **DB:** PostgreSQL.
- **Геоданные:** PostGIS.
- **Frontend:** Next.js, React или Vue.
- **Карта:** Mapbox GL JS или Leaflet.
- **Auth:** email + пароль, потом соцлогины.
- **Фоновая обработка:** Celery или RQ.
- **Кэш:** Redis.
- **Файлы:** S3-совместимое хранилище.

### Почему не Hugo как основа
Hugo подходит для лендингов, блога и страниц курсов, но не для личных кабинетов, платежей, карт, чатов и персональных профилей. Для твоего проекта логичнее сразу идти в динамический сайт. [stackoverflow](https://stackoverflow.com/questions/76386425/how-to-service-dynamic-content-with-hugo-on-demand)

## 6. Модель профиля пользователя

Профиль должен быть многослойным и включать не только тип личности, но и оси поведения, состояния и уровень приватности.

### Поля профиля
- Имя / псевдоним.
- Возраст.
- Пол.
- Страна / регион / псевдолокация.
- Тип личности.
- Темперамент.
- Интроверсия / экстраверсия.
- Открытость.
- Добросовестность.
- Кооперативность.
- Эмоциональная стабильность.
- Доминирование.
- Эмпатия.
- Конфликтность.
- Инициативность.
- Энергия.
- Текущий эмоциональный статус.
- Режим цели: дружба / отношения / работа / бизнес / квесты.
- Настройки приватности.
- Фото / аватар.
- Дополнительные интересы.
- Уровень / прогресс персонажа.

## 7. Логика совместимости

Совместимость должна считаться по нескольким уровням:
- тип личности,
- оси поведения,
- текущее состояние,
- цель взаимодействия,
- контекст локации,
- уровень приватности.

### Пример логики
- Для **отношений** важнее эмпатия, стабильность, согласие и совместимость по ценностям.
- Для **работы** важнее добросовестность, дисциплина, роль и устойчивость.
- Для **дружбы** важны энергия, интересы и эмоциональная открытость.
- Для **квестов** важны инициативность, открытость и скорость реакции.

## 8. Карта и регионы

Карта должна быть не только географической, но и мифологической. Реальные страны и города можно переименовывать в игровом стиле.

### Принцип
- Реальная страна может отображаться как легендарный регион.
- Пользователь сам выбирает, что показывать.
- Точные координаты не обязательны и должны скрываться по умолчанию.
- Можно использовать только область, район или псевдо-локацию.

### Пример
- Беларусь → «Княжество Тысячи Озёр».
- Минск → «Город Серебряных Площадей».
- Локации внутри страны → «Долина Тихих Озёр», «Площадь Белого Факела», «Сад Ночных Лилий».

## 9. Квесты и флэшмобы

Квесты должны быть безопасными и добровольными:
- участие только по согласию,
- без раскрытия точного адреса,
- с мягкой модерацией,
- без насильственного перевода на офлайн-встречи.

Подходящие форматы:
- образовательные квесты,
- социальные флэшмобы,
- групповые задания,
- совместные челленджи,
- локальные безопасные встречи.

## 10. Безопасность и приватность

Это один из главных блоков проекта. Для location-based social app privacy-critical. [enisa.europa](https://www.enisa.europa.eu/sites/default/files/publications/WP2017%20O-2-2-4%20GDPR%20Mobile.pdf)

### Обязательно
- Шифрование соединений.
- Раздельные уровни видимости данных.
- Скрытие точной геолокации по умолчанию.
- Блокировка и жалобы.
- Антиспам.
- Ограничение поиска по чувствительным данным.
- Логи модерации.
- Явное согласие на отображение данных.

## 13. Железо и запуск

На старте допустимо использовать твой ноутбук с Ubuntu как временный сервер для разработки и тестов, если нагрузка небольшая. Позже проект лучше перенести на VPS или облако. [dev](https://dev.to/jaspalch/building-a-home-server-0-hardware-linux-installation-and-initial-configuration-500i)

### Для начала достаточно
- Ubuntu.
- SSH.
- Docker.
- PostgreSQL.
- Nginx.
- Бэкапы.
- SSL-сертификат.

## 14. Итоговая рекомендация по реализации

Я советую делать проект как **динамическую платформу**, а не как чисто статический сайт. Публичная часть курсов может быть SEO-дружественной и предгенерированной, но ядро проекта должно быть полноценным приложением с backend, базой, картой, пользователями и платежами. [sanity](https://www.sanity.io/static-vs-dynamic-websites)
